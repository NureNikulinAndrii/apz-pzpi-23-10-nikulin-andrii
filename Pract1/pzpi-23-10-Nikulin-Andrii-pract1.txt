3 ОПИС ВИКОНАНОЇ РОБОТИ
3.1 Загальні відомості про шаблони (патерни) проєктування
Патерни проєктування — це перевірені, стандартизовані рішення типових проблем, що виникають під час розробки програмного забезпечення. Вони не є готовим кодом, а являють собою загальну концепцію або шаблон вирішення задачі, який можна адаптувати під потреби конкретної програми.
Класифікація GoF: За класифікацією «Банди чотирьох» (Gang of Four), патерни поділяються на три основні групи:
1. Породжувальні (Creational): відповідають за гнучке створення об'єктів (наприклад, Factory Method, Builder).
2. Структурні (Structural): допомагають складати об'єкти та класи у більші структури (наприклад, Adapter, Decorator).
3. Поведінкові (Behavioral): визначають алгоритми та способи взаємодії між об'єктами (наприклад, Observer, Strategy).
Область застосування: Об'єктно-орієнтоване проєктування та архітектура програмного забезпечення. Їх використовують для зменшення зв'язності (coupling) коду, підвищення його гнучкості, полегшення масштабування, підтримки та повторного використання існуючого коду.
3.2 Опис обраного патерна
• Назва патерна: Спостерігач (Observer)
• Тип шаблона (патерна): Поведінковий
• Основне призначення: Створення механізму підписки, який дозволяє одному об'єкту (суб'єкту/видавцю) повідомляти інші об'єкти (спостерігачів/підписників) про будь-які зміни свого стану. Патерн реалізує залежність типу «один-до-багатьох».
• Передумови застосування: Використовується тоді, коли зміна стану одного об'єкта вимагає автоматичної зміни інших об'єктів.
o Коли кількість цих залежних об'єктів невідома заздалегідь або може динамічно змінюватися під час виконання програми.
o Коли необхідно уникнути жорсткої зв'язності між класами (щоб суб'єкт нічого не знав про конкретні класи спостерігачів, а працював з ними через загальний інтерфейс).
• Переваги:
o Слабке зачеплення (Low Coupling): Видавець не залежить від конкретних класів підписників, що робить систему гнучкішою.
o Принцип відкритості/закритості (Open/Closed Principle): Можна додавати нові класи підписників без необхідності вносити зміни у код видавця.
o Можливість динамічно оформляти та скасовувати підписку під час виконання програми.
o 
• Недоліки:
o Витоки пам'яті: Якщо підписники не видаляються (не відписуються) після того, як стають непотрібними, збирач сміття не зможе видалити їх з пам'яті (проблема «забутого слухача» — Lapsed listener problem).
o Складність відлагодження: Оскільки виклики відбуваються неявно (через події), іноді буває важко відстежити потік виконання програми.
o Підписники сповіщаються у випадковому або непередбачуваному порядку (залежить від реалізації списку).
3.3 Структура шаблона (патерна)
• Опис структурних частин:
o Subject (Суб'єкт / Видавець): Інтерфейс або абстрактний клас, який містить список спостерігачів. Він надає методи для приєднання (attach()) та від'єднання (detach()) спостерігачів, а також метод notify() для їх сповіщення про зміну свого стану.
o Observer (Спостерігач / Підписник): Загальний інтерфейс для всіх об'єктів, які хочуть отримувати сповіщення від Суб'єкта. Зазвичай містить єдиний метод update(), який Суб'єкт викликає при зміні стану.
o ConcreteSubject (Конкретний суб'єкт): Клас, що реалізує інтерфейс Subject. Він містить бізнес-логіку та зберігає стан, який цікавить спостерігачів. Коли цей стан змінюється, об'єкт викликає успадкований або реалізований метод notify().
o ConcreteObserver (Конкретний спостерігач): Клас, що реалізує інтерфейс Observer. Він зберігає посилання на об'єкт ConcreteSubject, щоб мати змогу отримати його стан. Метод update() у цьому класі реалізує реакцію на сповіщення, синхронізуючи стан спостерігача зі станом суб'єкта.
• Опис взаємодії об’єктів
o Коли ConcreteObserver створюється або вирішує почати відстеження, він викликає метод attach() у ConcreteSubject, передаючи посилання на самого себе, і таким чином додається до списку підписників.
o Коли внутрішній стан ConcreteSubject змінюється (наприклад, оновлюються дані), він викликає свій метод notify().
o Метод notify() ітерується по всьому списку зареєстрованих підписників і викликає метод update() у кожного з них.
o Під час виклику update() ConcreteObserver отримує сигнал про зміну. Він може використати своє посилання на ConcreteSubject (або прийняти дані безпосередньо через параметри методу update()), щоб отримати актуальний стан суб'єкта та оновити власні дані чи інтерфейс.
o Коли спостерігачу більше не потрібні оновлення, він викликає метод detach(), видаляючи себе зі списку суб'єкта.

• UML-діаграми з поясненнями

Рисунок 1 - UML-діаграма
     Пояснення до діаграми: На діаграмі показано зв'язок типу агрегації (або композиції) між Subject та Observer: суб'єкт містить список об'єктів типу Observer (зв'язок "один-до-багатьох"). Завдяки тому, що Subject працює не з конкретними класами підписників, а з абстрактним інтерфейсом Observer, досягається слабке зачеплення. ConcreteSubject та ConcreteObserver є конкретними реалізаціями цих інтерфейсів. Стрілка від ConcreteObserver до ConcreteSubject вказує на те, що спостерігач знає про конкретного видавця і може запитувати в нього дані (наприклад, через метод getState()) під час свого оновлення.
3.4 Реалізація шаблона (патерна)
• Опис реалізації мовою програмування С#
• Основні етапи:
o Створення базових інтерфейсів
• Спочатку визначаємо контракти для Видавця (ISubject) та Підписника (IObserver). Видавець повинен вміти додавати, видаляти та сповіщати підписників, а підписник — оновлювати свій стан при отриманні сповіщення.
o Реалізація конкретних класів

o Демонстрація використання
• У головному методі програми створюємо об'єкти, підписуємо спостерігача на видавця і запускаємо логіку зміни стану, щоб побачити автоматичну реакцію.

2   using System.Collections.Generic;
3   
4   // Інтерфейс підписника (Спостерігача)
5   public interface IObserver
6   {

11  public interface ISubject
12  {
13      void Attach(IObserver observer);
14      void Detach(IObserver observer);
15      void Notify();
16  }
17

19  public class ConcreteSubject : ISubject
20  {
21      public int State { get; set; } = 0;
22      private List<IObserver> _observers = new List<IObserver>();
23  
24      public void Attach(IObserver observer)
25      {
26          _observers.Add(observer);
27      }
28  
29      public void Detach(IObserver observer)
30      {
31          _observers.Remove(observer);
32      }
33  
34      public void Notify()
35      {
36          foreach (var observer in _observers)
37          {
38              observer.Update(this);
39          }
40      }
